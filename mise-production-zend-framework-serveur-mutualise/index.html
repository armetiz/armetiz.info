Mise en production de Zend Framework sur un serveur mutualisé

Posted on 04/06/2010 · Posted in Zend

Lorsque l’on dispose d’un serveur dédié, la mise en production d’une application Zend Framework est assez simple; il suffit de modifier le DocumentRoot du VirtualHost.

Si l’on dispose d’un hébergement mutualisé, il faut aussi pouvoir proposer à nos visiteurs un accès au site via www.domain.tld et non pas www.domain.tld/public/

Première solution

La première solution, consiste à déplacer le fichier /public/index.php à la racine du domaine et de modifier la définition de la constante APPLICATION_PATH de ce même fichier.
Le seul soucis de cette solution, c’est que les appels aux ressources de type CSS ou JS devront être modifiés pour pointer vers www.domain.tld/public/
Personnellement, je trouve cela complètement nul…

Deuxième solution

La deuxième solution – bien plus élégante – passe par l’ajout d’un .htaccess à la racine du domaine.

RewriteEngine On
# Only apply to URLs on this domain
RewriteCond %{HTTP_HOST} ^(www.)?domain.tld$

# Only apply to URLs that aren't already under folder.
RewriteCond %{REQUEST_URI} !^/public/

# Don't apply to URLs that go to existing files or folders.
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d

# Rewrite all those to insert /folder.
RewriteRule ^(.*)$ /public/$1

# Also redirect the root folder.
RewriteCond %{HTTP_HOST} ^(www.)?domain.tld$
RewriteRule ^(/)?$ public/index.php [L]

Là, tout est magique et fonctionne sans aucune autre modification de l’application.

  19. Thomas Tourlourat
    30/07/2012 at 08:48 ·

    @André, la remarque n’est pas mauvaise du tout.
    Je vais répondre en prenant des pincettes, car je suis passé sous Symfony2 depuis plusieurs mois et j’ai un peu lâché Zend Framework.

    Le répertoire /application sous Zend Framework v1 contient la logique métier de votre application. Là où cela peux-être déroutant, c’est que /application contient aussi la partie vue de l’application : layout, helper…
    Maintenant, le répertoire /application n’a pas vocation a être rendu accesible sur http, c’est pour cela qu’il est conseillé de mettre les ressources HTTP dans le répertoire /public et non pas dans /application.

    Voici tout de meme une petite solution qui doit fonctionner.
    Prenons le cas de /application/views/assets qui est un répertoire nouvellement créé contenant les ressources HTTP de votre application. Vous pouvez faire un lien symbolique entre /public/assets -> /application/views/assets.
